# دراسة أنظمة الموارد البشرية وخارطة طريق SmartAttendance (ZYNORA HR)

> أُعدت 2026-07-17. الهدف: مقارنة النظام بالأنظمة المحلية والعالمية، وتحديد الفجوات، وخطة الذكاء الاصطناعي، وخطة تطبيق الموبايل، وإعادة تصميم صفحة EmployeeProfiles.

## 1. دراسة السوق: بمن نقارن أنفسنا

|---|---|---|

### الأنظمة الإقليمية (الشرق الأوسط — الأهم للمقارنة المباشرة)

|---|---|---|---|
| **MenaITech (MenaHR/MenaPay)** | الأقوى عراقياً وأردنياً | رواتب متوافقة مع القوانين المحلية، ضمان اجتماعي، عربي كامل | حساب الرواتب حسب قانون العمل العراقي (ضريبة، تقاعد 5%) ميزة حاسمة |
| **ZenHR** | الأردن/الخليج/العراق | واجهة عربية حديثة، خدمة ذاتية، تطبيق موبايل | RTL ممتاز + بساطة = يكسب العقود |

### ملاحظات السوق العراقي تحديداً
- أغلب الشركات العراقية بعدها على Excel أو أنظمة حضور بصمة معزولة — **ميزتنا التنافسية: ربط أجهزة البصمة + HR كامل + عربي/RTL أصيل**.
- المطلوب محلياً بالترتيب: (1) حضور/انصراف موثوق من البصمة، (2) رواتب حسب القانون العراقي، (3) إجازات وأرصدة، (4) عقود وإقامات للوافدين (موجود عدنا بلوحة التنبيهات)، (5) تقارير للإدارة.
- الإنترنت غير مستقر → التطبيق الموبايل لازم يتحمل offline جزئياً (تخزين مؤقت وإعادة إرسال).

## 2. تقييم SmartAttendance الحالي مقابل السوق

### موجود وقوي ✅
- حضور/انصراف + أجهزة بصمة + معالجة واستيراد وتصحيحات
- هيكل تنظيمي (شركات → فروع → أقسام) متعدد الشركات
- ملف موظف موسع (عقود، نهاية خدمة، إعادة تعيين، مدير مباشر)
- مركز وثائق + **لوحة تنبيهات وانتهاءات** (اكتملت 2026-07-17)
- صلاحيات ديناميكية + أدوار + تدقيق (AuditLogs)
- خدمة ذاتية وطلبات وموافقات، إعلانات واستبيانات
- شفتات، عطل، مخالفات وإجراءات تأديبية

### الفجوات مقابل المنافسين (بالأولوية) ❌
1. **الرواتب (Payroll)** — الفجوة الأكبر؛ MenaITech تكسب السوق بيها. يحتاج: راتب أساسي + بدلات + استقطاعات + ربط بالحضور (تأخير/غياب) + ضريبة وتقاعد عراقي + قسائم رواتب PDF.
2. **أرصدة الإجازات** — LeaveRequests موجودة بلا أرصدة (استحقاق سنوي، ترحيل، مرضية حسب القانون العراقي).
3. **Org Chart تفاعلي** من DirectManagerId (البيانات جاهزة).
4. **Onboarding/Offboarding checklists** — قوائم مهام مؤتمتة (عهدة، حسابات، وثائق مطلوبة) — نمط Personio.
5. **تقييم الأداء** — دورات تقييم بسيطة (أهداف + تقييم مدير) تكفي بالبداية.
6. **التوظيف (ATS)** — آخر الأولويات؛ السوق المحلي ما يطلبه بقوة.

## 3. خطة الذكاء الاصطناعي (المرحلة القادمة)

> المبدأ: AI يخدم مهام HR الفعلية، مو زر "AI" للديكور. التنفيذ عبر Claude API (نموذج `claude-sonnet-5` للعمليات، `claude-haiku-4-5` للمهام الخفيفة الرخيصة). يُبنى كخدمة منفصلة `SmartAttendance.AI` أو ضمن API الموجود، مع حارس: **كل مخرجات AI اقتراحات تُعرض للمستخدم، ما تُكتب بقاعدة البيانات مباشرة**.

### الترتيب المقترح (من الأسهل/الأعلى قيمة)
1. **مساعد HR محادثة (عربي)**: يجاوب من بيانات النظام — "منو غاب اليوم؟ شكد باقي بعقد فلان؟ شنو رصيد إجازاتي؟" — tool-use على استعلامات SQL جاهزة ومقيدة بصلاحيات المستخدم الحالي (نفس RoleSecurityMiddleware). أول ما يُبنى لأنه يلمس كل الوحدات ويبهر المستخدم.
2. **كشف شذوذ الحضور**: تحليل أنماط (تأخير متكرر أيام محددة، بصمات ناقصة مشبوهة، buddy punching محتمل) → تقرير أسبوعي للـ HR.
3. **قراءة الوثائق (OCR + استخلاص)**: رفع صورة جواز/هوية/إقامة → استخلاص الاسم والرقم و**تاريخ الانتهاء تلقائياً** (يغذي لوحة التنبيهات مباشرة). قيمة عالية جداً لمركز الوثائق.
4. **صياغة الإعلانات والكتب الرسمية**: توليد مسودات كتب إدارية/إعلانات بالعربية الرسمية من نقاط مختصرة (يركب على نظام الإعلانات الموجود).
5. **تنبؤ الاستقالات (لاحقاً)**: مؤشر خطورة من الحضور + المخالفات + مدة الخدمة — يحتاج بيانات تاريخية أكثر.

### بنية تقنية مقترحة
- جدول `AiConversations` / `AiAuditLogs` (كل استدعاء يُسجل: المستخدم، السؤال، الأدوات المستدعاة).
- مفتاح API بـ user-secrets/بيئة، ليس بالكود.
- حد شهري للاستهلاك لكل مستخدم.

## 4. خطة تطبيق الموبايل المحدود

> "محدود" = **خدمة ذاتية للموظف فقط**، مو لوحة أدمن. هذا بالضبط نمط Zoho People وZenHR الناجح.

### النطاق (الإصدار الأول)
- تسجيل دخول الموظف (نفس حسابات SystemUsers المرتبطة بـ EmployeeId)
- بصمة حضور/انصراف من الموبايل بـ **GPS geofencing** (ضمن نطاق الفرع) + صورة سيلفي اختيارية
- عرض سجل حضوري الشهري + بصماتي الناقصة
- تقديم طلب إجازة/استئذان + متابعة حالته
- رصيد إجازاتي (بعد بناء نظام الأرصدة)
- الإعلانات والإشعارات (انتهاء وثائقي، الموافقات)
- ملفي الشخصي (عرض فقط + طلب تعديل)

### الاختيار التقني
| الخيار | التقييم |
|---|---|
| **PWA (تحويل صفحات EmployeePortal الموجودة)** | ✅ **ابدأ بيه**: أسرع طريق، بدون متجر، يشتغل على كلشي. يكفي للنطاق أعلاه. GPS والكاميرا متاحة بالمتصفح. حدوده: إشعارات push على iOS أضعف |
| **Flutter** | الخيار الثاني إذا احتجنا native حقيقي (bluetooth للأجهزة، offline قوي) — فريق واحد لـ Android/iOS |
| **.NET MAUI** | نفس لغة المشروع (C#) لكن نضجه أقل من Flutter بالواجهات العربية RTL |

**التوصية**: مرحلة 1 = PWA فوق `SmartAttendance.API` الموجود (نبني endpoints للخدمة الذاتية + JWT). مرحلة 2 = Flutter إذا ثبت الطلب. بهذا الترتيب ما نخسر شغل: نفس الـ API يخدم الاثنين.

### متطلبات مسبقة على الـ API
- JWT authentication (الويب حالياً كوكيز) + refresh tokens
- Endpoints: `POST /api/attendance/punch` (مع lat/long والتحقق من geofence), `GET /api/me/attendance`, `POST /api/leave-requests`, `GET /api/me/notifications`
- Rate limiting + تسجيل الجهاز (device binding لمنع بصمة الغير)

## 5. إعادة تصميم صفحة EmployeeProfiles («عمليات الموارد البشرية»)

### التشخيص (مؤكد من الكود)
`Pages/EmployeeProfiles/Index.cshtml` = **1102 سطر** + code-behind 894 سطر، تدمج **5 أنظمة مستقلة بصفحة وحدة** مع 9 معالجات POST:
1. الإعلانات (إنشاء/قوالب/معاينة/أرشفة/تفعيل)
2. الاستبيانات والاستطلاعات/الانتخابات (إنشاء/حذف/تفعيل)
3. نبض الموظفين والملاحظات (رد/إغلاق)
4. التقدير والإنجازات
5. الحملات والتعاميم

هذا سبب إحساس "الهوسة": ما في تسلسل بصري، كل الأقسام تتنافس على الشاشة، والمستخدم ما يعرف من وين يبدأ.

### الحل المقترح: تفكيكها إلى مركز اتصالات داخلية (نمط BambooHR/HiBob)
```
/Engagement            ← صفحة hub خفيفة: 4 بطاقات كبيرة + أرقام ملخصة فقط
/Engagement/Announcements  ← الإعلانات (قائمة + إنشاء + قوالب)
/Engagement/Polls          ← الاستبيانات والانتخابات
/Engagement/Feedback       ← نبض الموظفين والملاحظات
/Engagement/Recognition    ← التقدير + الحملات (يدمجان، الأصغر حجماً)
```
- كل صفحة ≤ 300 سطر، معالجاتها فقط، ونمط `hrms-page` الموحد (نفس لوحة التنبيهات).
- الـ hub يعرض: عدد الإعلانات الفعالة، استبيان مفتوح، ملاحظات بلا رد، آخر تقدير — وكل بطاقة تفتح صفحتها.
- التنفيذ تدريجي بدون كسر: تُنشأ الصفحات الجديدة وتُنقل المعالجات، وتصير `/EmployeeProfiles` redirect للـ hub (الروابط القديمة بالقائمة والإشعارات تظل شغالة).
- تقدير الجهد: جلستان — الأولى (hub + الإعلانات + الاستبيانات)، الثانية (الملاحظات + التقدير + التحويلات وحذف القديم).
