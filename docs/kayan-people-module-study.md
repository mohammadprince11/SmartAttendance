## بنية المودل (خريطة عليا)

مودل أشخاص = **ملف موظف 360°** + **عمليات م.ب.** + **إعدادات (بيانات مرجعية)** + أقسام مساندة.

روابط رئيسية مكتشفة:
- `/Employees/Dashboard/Index` — رسومات بيانية
- `/Employees/Home/Index` — عرض/قائمة الموظفين + ملف الموظف
- `/Employees/OrganizationalStructures/Index` — الهياكل التنظيمية
- `/Employees/Reports/TabularReports` — التقارير
- `/Employees/CustomRequestsScreening/Index` — متابعة الطلبات المخصصة
- `/Employees/AssetsManagement/Index` — إدارة العهد
- `/Employees/TemporaryHeadofUnitAllocation/Index` — تخصيص رئيس وحدة مؤقت
- `/ConnectedApps/Configuration/Index` — التطبيقات المتصلة

### القائمة الفرعية الكاملة (من شجرة التنقّل)

**تفاعل الموظفين**
- `/Employees/Announcement/Index` — الإعلانات
- `/Employees/VotesAndPolls/Index` — الإنتخابات واستطلاعات الرأي
- `/Employees/SuggestionsComplaintsBox/...` — اقتراحات وشكاوى
- `/Employees/WallPostsSetup/ViewPostsIndex` — متابعة منشورات الحائط

**تحديثات الموظف** (تعديلات جماعية مصنّفة — `/Employees/ChangeTransactions/Index?Type=...`): معلومات أساسية · معلومات التوظيف والتخصيص · معلومات الاتصال · معلومات الحضور · معلومات مالية · معلومات الدفع · حقول إضافية. → **هذي التصنيفات السبعة تكشف مجموعات حقول كيان الموظف.**

**عقود الموظفين**: `ContractsManagement/ViewContracts` (العقود) · `ContractsManagement/Index` (تحديثات العقود)

**متابعة الطلبات المخصصة**: `/Employees/CustomRequestsScreening/Index`

**حالات المخالفات**: `/Employees/ViolationCases/TabIndex` · `/Employees/ViolationActions/Index` (إجراءات تأديبية)

**تخصيص رئيس وحدة مؤقت**: `/Employees/TemporaryHeadofUnitAllocation/Index`

## 2) اللوحات الفرعية بملف الموظف (سجلّات متعدّدة لكل موظف)

كلها علاقة **1‑إلى‑متعدد مع الموظف** (جدول مستقل، FK = EmployeeId):

| اللوحة | الكيان المقترح | حقول رئيسية (متوقّعة/مؤكّدة) |
|---|---|---|
| العائلة | `EmployeeDependent` | نوع القرابة (شريك/إبن/بنت/قريب) · الاسم · تاريخ الميلاد · الجنس · معال (نعم/لا) · ملاحظات |
| التعليم | `EmployeeEducation` | المؤهل/الدرجة · التخصص · الجهة/الجامعة · سنة التخرج · المعدّل |
| الخبرات | `EmployeeExperience` | جهة العمل · المسمى · من–إلى · سبب الترك |
| الشهادات | `EmployeeCertificate` | اسم الشهادة · الجهة المانحة · تاريخ الإصدار/الانتهاء |
| دورات تدريبية | `EmployeeTraining` | اسم الدورة · الجهة · من–إلى · النتيجة |
| الملفات الطبية | `EmployeeMedicalRecord` | نوع · تاريخ · ملاحظات · مرفق |
| العنوان | `EmployeeAddress` | دولة/مدينة · الحي/الشارع · نوع (دائم/حالي) |
| الكفاءات | `EmployeeCompetency` | كفاءة (FK master) · المستوى/التقييم |
| العهد | `EmployeeAsset` | ← مفصّل بقسم العمليات |
| نتائج التقييم | `EmployeeEvaluationResult` | ← مفصّل بقسم العمليات |
| إدارة الإقامة | `EmployeeResidency` | (للوافدين) رقم إقامة · إصدار/انتهاء · كفيل · مهنة · نوع تأشيرة |

**تبويبات الملف**: شخصي · مالي · الملفات الإلكترونية (وثائق) · سجل الموظف (Audit) · الإجراءات التأديبية · إدارة الإقامة. + عدّاد **فترة تجربة** حي على البطاقة.

## 3) عمليات الموارد البشرية — كيانات مؤكّدة بالفحص

### إدارة العهد `EmployeeAsset` (`/AssetsManagement`)
EmployeeId · **نوع العهدة** (كمبيوتر/هاتف/سيارة — master `AssetType`) · الاسم/الموديل · FromDate · ToDate · المبلغ + العملة · **متناقص** (bool depreciating) · **موافقة الموظف** (لم ترسل/تمت الموافقة — workflow) · (متوقّع: رقم تسلسلي، ملاحظات، مرفق).

### العقود `EmployeeContract` (`/ContractsManagement`) — **كيان مستقل متعدّد لكل موظف**
EmployeeId · رقم العقد · **نوع العقد** (محدود N سنة / غير محدود — master) · FromDate · ToDate · (محسوب: الأيام المتبقية). ← عدنا حالياً حقول عقد على الموظف؛ كيان يعطي **تاريخ عقود** ويغذّي لوحة الانتهاءات.

### إنهاء الموظف `EmployeeTermination` (`/EmployeeTermination`) — workflow 3 مراحل
مراحل: إنهاء الخدمات → تسوية الإنهاء → منجز. حقول: **رقم مرجعي** (EX26‑xx) · تاريخ الإيقاف · **سبب الإيقاف** (انتهاء عقد/اختياري — master) · فترة الإنذار · تاريخ انتهاء الإنذار · **آخر يوم عمل** · المصدر (اشخاص/الرواتب) · (تسوية = مكافأة نهاية خدمة/مستحقات).

### نتائج التقييم `EmployeeEvaluationResult` (`/EvaluationsScreening`)
EmployeeId · سنة · فترة (من–إلى) · النتيجة % · النتيجة كتقدير (رقم) · النتيجة بالكلمات · **المصدر** (مباشر / إدارة الأداء). ← يُدخل يدوياً أو يُغذّى من مودل الأداء.

### مهام الموظفين `EmployeeProcess`/`Task` (`/CompanyProcesses/EmployeesTasks`) — محرّك عمليات
العنوان · بدأ بواسطة · تاريخ الإنشاء · **بخصوص** (موظف مستهدف) · الحالة (مسودة/قيد الانتظار/منجز). قوالب: On‑Boarding, Appraisal, مشتريات... = **checklists/workflow قابلة للتعريف** بخطوات.

### الطلبات المخصّصة `CustomRequest` (`/CustomRequestsScreening`)
أنواع طلبات خدمة ذاتية **قابلة للتعريف**، شاشة متابعة تُفلتر بـ: وحدة عمل · هيكلية · موقع · **مجموعة الموظفين** · **كفيل** · حالة. حقول التقرير: اسم الطلب · تاريخ · حالة.

### حالات المخالفات (`/ViolationCases`) + الإجراءات التأديبية (`/ViolationActions`)
≈ موجود عدنا (Violations/Disciplinary).

### تخصيص رئيس وحدة مؤقت (`/TemporaryHeadofUnitAllocation`)
تفويض إداري مؤقت لعقدة بالهيكل (بديل عند إجازة المدير).

## 4) الهياكل التنظيمية — **ثلاث هياكل متوازية** (مهم!)
- **هيكلية الشركة**: شجرة وحدات بأنواع (**كيان قانوني / فرع رئيسي / فرع فرعي**) + عدّاد موظفين.
- **الهيكل الهرمي**: تبعية إدارية (من يرفع لمن) — منفصل عن الوحدات.
- **الهيكل الوظيفي**: مسميات/وظائف.

مقابل عدنا: عندنا Company→Branch→Department ثابت + org chart من DirectManagerId. كيان يفصل الوحدة القانونية عن التبعية الإدارية عن الوظيفة → مرونة أعلى.

## 5) البيانات المرجعية (Master Data / Lookups)

مستنبطة من قوائم شاشة الموظف + قائمة «إعدادات الموارد البشرية»:

**قوائم كيان الموظف**: الجنس · الحالة الاجتماعية · الديانة · الجنسية · الدول · المدن · نوع العقد · نوع العمل · طبقة العمل (Grade) · سلم الرواتب · ملف الضريبة · شريحة الضمان الاجتماعي · مخطط رمز الموظف · نوع العهدة · سبب الإيقاف · الكفاءات · مجموعة الموظفين · الكفلاء · مراكز التكلفة · البنوك/طرق الدفع.

**قائمة إعدادات م.ب.**: التهيئة (المرجعيات) · طلبات الموظفين المخصّصة · إعدادات مهام الموظفين · إعدادات المخالفات · فترة التجربة · فترة الإنذار · إعدادات الخدمة الذاتية · **إقرارات الموظفين** (توقيع سياسات) · أسباب الإيقاف · إعدادات الملفات الإلكترونية · مركز الإشعارات · **حقول إضافية مخصّصة (custom fields)**.

> عدنا يطابق: فترة التجربة، فترة الإنذار، أسباب الإيقاف، الخدمة الذاتية، مركز الإشعارات. ناقص عدنا: كتالوج مرجعيات موحّد (ديانة/جنسية/طبقة/سلم رواتب/نوع عهدة/كفيل/مجموعة موظفين)، الحقول المخصّصة، الإقرارات.

## 6) خريطة العلاقات (ERD مبسّط)

```
Company/BusinessUnit (شجرة: كيان قانوني→فرع رئيسي→فرع فرعي)

        ├── Location (موقع)
```

## 7) المخطط المقترح لـ SmartAttendance (ما نبنيه)

الأولوية العليا (تكمّل ملف 360° وأقرب لبيانات موجودة):

1. **EmployeeDependent** (العائلة/المعالين) — أصغر مكسب، عالي القيمة (يخدم الرواتب/التأمين لاحقاً). حقول: EmployeeId, RelationType(enum), Name, BirthDate, Gender, IsDependent(bool), Note.
2. **EmployeeEducation / EmployeeExperience / EmployeeCertificate** — سجلات السيرة. نمط موحّد (EmployeeId + عنوان + جهة + تواريخ).
3. **EmployeeAsset** (العهد) — EmployeeId, AssetType(FK/enum), Name/Model, Serial?, FromDate, ToDate?, Amount, Currency, IsDepreciating, EmployeeApprovalStatus, Note. + شاشة تُغذّي «الاسترجاع عند الإنهاء».
4. **EmployeeContract** ككيان تاريخ مستقل (بدل حقول العقد على الموظف) — يغذّي لوحة الانتهاءات، مع migration ينقل الحقول الحالية.
5. **EmployeeResidency** (الوافدون) — رقم إقامة, إصدار, انتهاء, كفيل, مهنة, نوع تأشيرة. يغذّي لوحة التنبيهات (عدنا فيزا كوثيقة فقط).
6. **EmployeeTask/Process** — checklists تعيين/إنهاء (قوالب + خطوات + مسؤول + حالة).
7. **EmployeeEvaluationResult** — تخزين نتائج تقييم بسيطة (تمهيد لمودل الأداء).

بيانات مرجعية مشتركة مطلوبة: `Nationality, Religion, MaritalStatus, JobGrade, SalaryScale, AssetType, Sponsor, EmployeeGroup, CostCenter` — جدول lookup موحّد أو enums حسب الثبات.

قرارات معمارية:
- الملف 360° = جداول فرعية 1:N، كلها EF entities + `HrmsDatabase` schema idempotent (نفس نمط المشروع).
- الهيكل: نبقى على Company→Branch→Department حالياً (تحويله لشجرة أنواع = مشروع كبير، مؤجّل).
- كل كيان جديد: نفس نمط الطبقات (Domain entity + Configuration + DbSet + schema class + صفحات Razor + رابط قائمة + RoleSecurityMiddleware).

## 8) كتالوج الكيانات الكامل (من شاشة التقارير — مرجع نهائي)

كل «تقرير نظام» = جدول بيانات. القائمة تؤكّد الكيانات وتكمّلها:

معلومات شخصية · معلومات حسب التاريخ (history) · أسماء · معلومات عمل · معلومات اتصال · **كفاءات** · **خبرات** · **شهادات** · في مرحلة الإنهاء · منتهية خدماتهم · **تعليم** · **دورات تدريبية** · **ملفات طبية** · **وثائق** · **عهد** · **عناوين** · **شركاء (أزواج)** · **أقرباء** · **أبناء** · **نتائج تقييم** · **أجر متغيّر مرتبط بالأداء** · تعديلات (audit) · تحديثات مخصّصة · في فترة التجربة · **جهات اتصال الطوارئ** · مخالفات · إجراءات تأديبية · اقتراحات وشكاوى · **إقرارات الموظفين** · عدد الموظفين حسب الموقع · مهام.

**كيانات جديدة اكتُشفت هنا** (تُضاف للمخطط):
- `EmployeeEmergencyContact` — جهة اتصال طوارئ (اسم، صلة، هاتف).
- العائلة مقسّمة: شركاء/أقرباء/أبناء ← يمكن جدول `EmployeeDependent` واحد بـ RelationType، أو ثلاثة. كيان يفصلها بالتقارير لكن الأرجح جدول واحد.
- `PerformancePay` — أجر متغيّر يربط نتيجة التقييم بالراتب.
- التقارير فيها **٣ تبويبات**: تقارير النظام · **تقاريري** (باني تقارير مخصّص) · تمت مشاركتها معي (مشاركة تقارير).

## 9) آلية الواجهة (السلايد / الفتح-الإغلاق / التوزيع) — لعكسها على نظامنا

النظام مبني على إطار **BT-view** (SPA) فوق **Kendo UI Mobile** + **Bootstrap grid**.

### أ. الشاشات المنزلقة (السلايد) — نمط «شاشات مكدّسة» push/pop
- كل شاشة = `.BT-view-contanier` فيها هيدر `.BT-view-header` (**`position:fixed; z-index:99`**، ارتفاع 45px) + جسم `.BT-view-body`.
- **الفتح (open)**: `$.BT_load_view()` → `$.BT_show_loadedView()` — يدفع view جديد فوق الحالي.
- **الإغلاق (close/الكروس)**: زر الرجوع `<a class="BTIcon icon-back" onclick="$.BT_unload_level()">` (السهم → أعلى الهيدر) — يسحب الـ view العلوي.
- **الحركة**: Kendo `k-fx-slide` — الشاشة الجديدة تدخل من `transform: translateX(100%)` إلى `0`، والقديمة تُدفع لـ `translateX(-100%)`، بـ **`transition: 350ms ease-out`**. الرجوع يعكس الاتجاه.

### ب. الدرج الجانبي (aside) — سلايد جانبي بعرض قابل للضبط
- دوال: `$.BT_setwidth_asideView()` (يحدّد العرض) · `$.BT_unload_asideView()` (إغلاق).
- يُستعمل للفلاتر/التفاصيل الجانبية. زر الإغلاق X = `<span class="BTIcon-btn-close">`.

### ج. Popup/Modal
- `$.BT_unload_popup()` للإغلاق. (شاشة «إضافة موظف» فُتحت هكذا: modal بتبويبات + أزرار حفظ/إلغاء بأسفل ثابت.)

### د. توزيع اللوحات (الـ layout distribution)
- الحاوية `.row` (`display:flex; flex-wrap:wrap`).
- كل لوحة داخل عمود **`col-md-12 col-lg-6 col-xl-4`**:
  - تابلت (md): عمود واحد (كامل العرض)
  - ديسكتوب (lg): عمودين (نصف)
  - عريض (xl): **٣ أعمدة** (ثلث)
- البطاقة نفسها `.emp-box` (~325px). البطاقات الملخّصة تعرض أيقونة + «لا يوجد بيانات» فارغة، أو محتوى مكثّف.

### هـ. كيف نعكسها على SmartAttendance (Razor Pages، server-rendered)
- **التوزيع**: نفس Bootstrap grid `row > col-md-12 col-lg-6 col-xl-4` (أو Grid بـ hrms-core.css) — سهل ومطابق.
- **السلايد/الدرج**: بما إننا server-rendered، الأنسب:
  - **Bootstrap 5 Offcanvas** جاهز (درج ينزلق من الجانب مع X للإغلاق) لللوحات الجانبية/الفلاتر/التفاصيل السريعة.
  - أو CSS بسيط: لوحة `position:fixed` بـ `transform:translateX(100%)` تُبدّل لـ `translateX(0)` مع `transition: transform .35s ease-out`، وزر X يشيل كلاس `open` (يحاكي `BT_unload`).
  - **View Transitions API** لانتقالات الصفحات الكاملة (سلايد بين الصفحات) بدون SPA.
- **الهيدر الثابت**: شريط علوي `position:sticky/fixed` بعنوان + زر رجوع (→) + أزرار إجراءات — مطابق لنمطنا `page-header`.
- **التوصية**: لملف الموظف 360°، نعتمد **صفحة ملف واحدة + تبويبات + شبكة بطاقات (offcanvas للتفاصيل/الإضافة السريعة)** — يعطي إحساس كيان بدون إعادة كتابة كـ SPA.

## 12) صفحة التقارير — المواصفات الكاملة (جولة 4، فحص عميق)

### أ. الصفحة الرئيسية
- 3 تبويبات: **تقارير النظام · تقاريري · تمت مشاركتها معي** + بحث.
- العرض: **شبكة بطاقات 3 أعمدة** (`.row > .col-md-4 > .BT-report-card`)، كل بطاقة: **أيقونة + اسم التقرير + زر تثبيت (pin)**. النقر يفتح `view_report('<enc>')` (BT-view مدفوع).
- تبويب «تقاريري» الفارغ: empty state (أيقونة + رسالة + **زر أخضر «تقرير جديد»** → `ViewReportType()`).

### ب. شاشة تشغيل التقرير
- هيدر: **رجوع** (`BT_unload_level`) + **طباعة** (`KY_pq_PrintAsPDF`) + **تصدير إلى إكسل** (`KY_pq_ExportToExcel`) + **بحث** (`asidefilter` — يفتح درج جانبي).
- الجدول: ParamQuery grid (`KY_pq_*`) بفلاتر لكل عمود + تمرير أفقي + ترقيم أسفل.
- **درج الفلاتر الجانبي** (`BT-asideInnerView-body`): حقول حسب التقرير — رمز الموظف (نص) + الاسم الكامل (نص) + **وحدة عمل multi-select بشجرة + «اختيار الكل»**.

### ج. باني التقارير (تقرير جديد)
1. **مودال نوع التقرير**: «تقارير مجدولة» أو «تقارير محورية (Pivot)» → استمر.
2. صفحة الباني (BT-view كامل + زر **حفظ** أخضر بالهيدر):
   - اسم التقرير (عربية + لغة أخرى) + **الوصف**.
   - **من يمكنه مشاهدة هذا التقرير**: `SecurityTypeID` = الجميع / أنا فقط / مستخدمين محددين (+ قائمة مستخدمين) + شيكات: مشاركة مع المديرين عبر الخدمة الذاتية / مع الموظفين.
   - شيك «إخفاء الأعمدة ذات القيم الصفرية».
   - **اختيار الأعمدة**: قائمة فئة (`ReportFieldsCategoryID`) → تملأ «الحقول المتاحة» (`lstFields`) → تُنقل لـ«الحقول المحددة» (`lstselectedFields`) — dual listbox بسحب وإفلات.
   - **اختيار المرشحات**: نفس النمط (`ReportFilterFieldsCategoryID` → `lstfilterFields` → `lstselectedfilterFields`).
3. **فئات الحقول (24)**: شخصية · مالية · مدير مباشر · جهة طوارئ · كفاءات · خبرات · شهادات · تعليم · دورات · طبية · وثائق · عهد · عناوين · شركاء · أبناء · أقارب · نتائج تقييم · أجر متغير · مخالفات · تحديثات الموظف · معلومات الشركة · **آخر تعليم · العنوان الحالي · البنك الرئيسي** (مشتقات "أحدث سجل").
4. فئة «الشخصية» وحدها = **144 حقل** (كل حقل بنسختين English/العربية).
